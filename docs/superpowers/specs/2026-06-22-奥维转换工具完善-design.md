# 奥维数据格式转换工具完善 — 设计文档

日期：2026-06-22
状态：待实现

## 1. 背景与目标

现有工具把奥维 `.ovkml` / `.ovobj` 转换为 KML / SHP / DXF。经实测发现一批确认的问题，其中坐标系处理是核心隐患。本次目标：

1. **引入真实坐标系转换能力**（核心），覆盖 WGS84 / GCJ02 / BD09 / CGCS2000。
2. **修复所有已实测确认的问题**：OVOBJ 文件夹分组错误、SHP 中文乱码与分组丢失、DXF 规范性、KML 编码声明。
3. 不引入任何新依赖（转换纯 Python），不更换 UI 框架（保持 Tkinter），不动 `6ca3b-main/` 参考二进制。

## 2. 已确认的问题清单（实测依据）

## 3. 架构调整：引入转换管线

当前转换逻辑内联在 `ui/main_window.py:_do_convert`，写入器各自处理坐标系标注。引入一层编排，坐标转换集中一处：

```
解析 Parser → GeoDocument → [坐标转换 transform_document] → 写入器 Writer
```

- 坐标转换**只做一次**，产出坐标已转换的新 `GeoDocument`，三个写入器保持"无脑写出"。
- 新增 `convert/conversion_service.py`，提供 `convert_file(filepath, target_crs, ovobj_src_crs, formats, out_dir, sibling_files=None)`，把"解析→源坐标系判定→转换→多格式写出"的编排从 UI 抽出，UI 只负责收集参数与显示进度。便于测试与复用。

### 3.1 源坐标系判定（导入后自动/手动）

坐标系数值上无法从坐标本身反推（同一点 WGS84 与 GCJ02 仅差数百米，数值同样合理），因此按格式区别对待：

| 格式 | 源坐标系策略 |
|------|-------------|
| **OVKML** | **自动检测**：读 `<OvCoordType>`（每对象级，取文档首个有效值作文档级）。解析后 `GeoDocument.coord_type` 即为检测结果，UI 显示"检测到：XXX"，无需用户干预。 |
| **OVOBJ** | 用户在 UI 指定（默认 `CGCS2000`）。**同名兜底**：若导入列表中存在同名 `.ovkml`（如 `X.ovobj` + `X.ovkml`），自动解析该 OVKML 的 `OvCoordType` 作为 OVOBJ 源坐标系，优先级高于 UI 默认值。 |

- `conversion_service.convert_file` 接收 `sibling_files`（本批次全部导入文件路径）以实现同名兜底匹配（按 `Path.stem` 比对）。
- OVOBJ 二进制未解码出坐标系字段，且数值不可反推——此处不做"自动识别"，只做"已知则借用"。若将来获得同一数据导出为不同坐标系的多份 OVOBJ 样例，可通过字节 diff 定位坐标系字段，届时再实现 OVOBJ 自动检测（本次不做）。

## 4. 坐标转换模块 `transforms/coordinate_transform.py`

纯 Python、零依赖。实现业界标准算法。

### 4.1 支持的坐标系（`CoordType` 枚举新增 `BD09`）

```
WGS84  ⇄  GCJ02  ⇄  BD09
  ‖ (≈ 等同，恒等变换)
CGCS2000
```

- **CGCS2000 ⇔ WGS84**：测图尺度差异厘米级，按恒等处理。
- **WGS84 ⇄ GCJ02**：火星坐标加偏算法；逆变换（GCJ02→WGS84）用迭代逼近（粗纠偏，精度约 1m）。
- **GCJ02 ⇄ BD09**：百度官方 BD09ll 算法。
- 任意两系互转通过以 GCJ02 为枢纽（或 WGS84）组合实现。

### 4.2 关键算法点

- **中国境外判断**：GCJ02 偏移仅在中国境内有定义。`out_of_china(lon, lat)` 为真时，所有涉及 GCJ02/BD09 的转换返回原值（不偏移），并由调用方记录一次告警。
- **BD09 仅处理 BD09ll（经纬度）**，不处理 BD09mc（墨卡托平面）。
- **高程 alt 不变**，只转换经纬度。

### 4.3 接口

```python
def transform_point(lon: float, lat: float, src: CoordType, dst: CoordType) -> tuple[float, float]
def transform_document(doc: GeoDocument, target: CoordType) -> GeoDocument  # target=UNKNOWN 表示"与输入坐标系一致"(不转换)
```

- `transform_document`：当 `target == UNKNOWN`（"与输入坐标系一致"，即不转换）时原样返回，输出坐标系即各对象/文档检测到的源坐标系；否则对每个对象按其 `coord_type`（源）转到 `target`，并把结果 `coord_type` 标为 `target`、文档 `coord_type` 标为 `target`。
- 源坐标系为 `UNKNOWN` 的对象：回退使用文档级 `coord_type`；仍未知则跳过转换并告警。

## 5. OVOBJ 解析修复（P1/P2/P3）

核心是用文件里**已有的真实顺序信息**替代平均分配：

- **统一线性扫描**：在一次字节扫描中同时发现"对象记录"和"文件夹名标记"，各自记录其字节偏移。
- **就近归属**：每个对象归属到其偏移之前**最近出现的文件夹名标记**；首个文件夹标记之前的对象归入默认文件夹（文档名）。
- **收紧文件夹名识别**：候选文件夹名须不与任何已解析对象记录的字节区间重叠，降低把对象名/属性串误判为文件夹的概率。
- **诚实标注局限**：OVOBJ 二进制格式未完全逆向，线/面对象缺样例。保留坐标范围合理性校验作为兜底，但在代码注释与文档注明这是启发式解析。
- **回归基线**：以同名 OVKML 为真值，断言 B线含 {B4, B6-1}、A线含 A5、各文件夹对象集合与 OVKML 一致。

## 6. 写入器修复

### 6.1 SHP（P5/P6/P7）
- 生成 `.cpg` 文件，内容 `UTF-8`；DBF 以 UTF-8 编码写出。
- 属性表新增 **`FOLDER`** 字段（C, 100），写入对象所属文件夹名 → 保留线路分组。
- 多边形：写出前确保外环**顺时针**（按有向面积判断，逆时针则反转）。
- `.prj` 按**转换后的目标坐标系**选择 WKT 模板（已有 WGS84/CGCS2000/GCJ02 模板，新增 BD09 用 WGS84 椭球模板兜底）。

### 6.2 DXF（P8）
- 写出 `$ACADVER`（AC1009/R12 兼容档）与必要的 HEADER 变量。
- 在 `TABLES` 段写出 **`LAYER` 表**，先定义所有用到的图层（按文件夹名），再在 ENTITIES 引用。
- 保留 UTF-8 输出；中文图层名的 CAD 兼容性边界在文档注明（不同 CAD 对 DXF 编码处理不一，必要时用户可改用 SHP）。

### 6.3 KML（P9/P10）
- 输出补 `<?xml version="1.0" encoding="UTF-8"?>` 声明。
- `OvCoordType` 标注为**转换后的目标坐标系**。
- （P10，小修）OVKML 解析 MultiGeometry 时一并支持 LineString/Polygon，不再只取 Point。

## 7. UI 增强（`ui/main_window.py`）

- 新增 **"目标坐标系"** 下拉：`与输入坐标系一致(默认) / WGS84 / CGCS2000 / GCJ02 / BD09`。
  - **"与输入坐标系一致"** = 不做坐标转换，输出坐标系等于自动检测/手选的源坐标系（OVKML 用检测值，OVOBJ 用兜底/手选值）。映射为 `transform_document` 的 `target=UNKNOWN`。
- 新增 **"OVOBJ 源坐标系"** 下拉：默认 `CGCS2000`，仅在转换 OVOBJ 且无同名 OVKML 兜底时生效。OVKML 源坐标系自动检测，不需用户设置。
- 转换调用改为走 `conversion_service`，把每个文件实际采用的源坐标系（检测/兜底/手选）与境外坐标告警汇总到状态栏/完成提示，便于用户核对。
